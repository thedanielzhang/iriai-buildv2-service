Existing vs Proposed Execution Flow

A single comparison surface for the full develop workflow: unchanged planning, the current `ImplementationPhase` / `_implement_dag` monolith, the proposed transactional execution control plane, and preserved post-execution flows. Use the toggle to isolate existing behavior, proposed behavior, or the exact architectural deltas.

Flow comparison mode Aligned by execution stage, not by source file.
Primary change

Authority moves out of the monolith

Existing execution centralizes scheduling, dispatch, verification, repair, commit, checkpoint, and post-DAG gates inside the implementation phase. The proposed architecture assigns each authority to a typed, durable module with explicit proof boundaries.

Existing

Artifacts carry mixed meaning

`dag-task:*`, `dag-verify:*`, and `dag-group:*` are durable evidence, but current consumers can treat latest artifacts as more authoritative than their provenance proves.

Proposed

Typed rows become canonical

Typed journal rows, evidence graphs, queue rows, and gate proofs become the source of truth. Legacy `dag-*` artifacts remain synchronous compatibility projections while readers migrate.

Scoping Phase Flow

`ScopingPhase` remains a preserved planning phase. It establishes the feature boundary before decomposition and executable planning begin.

What changes

S1. Resume And Recovery Check

Determines whether approved scope already exists.

Existing

Approved marker and draft artifacts

The phase reads the scope approved marker, `scope`, structured scope draft, and project context. If approved scope exists, it finalizes scoping outputs or recovers a scope model from the approved text.

Proposed

Scope remains upstream evidence

No execution-control state is created here. The approved scope later becomes an input to task contracts, verifier context, and planning traceability.

Preserved

No authority change

Resume/recovery behavior stays in planning. The control plane must consume scope, not reinterpret or bypass it.

S1A. Input Hydration

Loads scope text, structured draft, and project context.

Existing

`scope`, `_SCOPE_DRAFT_KEY`, project context

Before deciding whether to interview, the phase hydrates persisted scope text, structured `ScopeOutput` draft state, and project context so resume paths have the same inputs as fresh paths.

Proposed

Hydrated intent remains immutable upstream context

Execution modules later consume this context through contracts and verifier packages. They do not ask scoping to rerun or reinterpret partially hydrated state.

Preserved

Recovery stays planning-local

Scope recovery can influence planning, but it does not create a runnable task, sandbox, queue item, or checkpoint.

S2. Scope Interview

Collects focused feature boundaries from the user.

Existing

`HostedInterview` with `scoper`

If no draft exists, `ScopingPhase` asks about repos involved, change type, and out-of-scope areas, producing a structured `ScopeOutput` and hosted `scope` artifact.

Proposed

User interaction remains unchanged

The control plane does not replace the scoping interview. Later execution contracts should cite the approved scope when enforcing non-goals and allowed surfaces.

Preserved

Intent capture stays human-led

This phase remains focused on feature intent, not product-file mutation or execution scheduling.

S2A. Draft Persistence

Stores structured scope before approval.

Existing

Structured draft written before the gate

Fresh interviews write the draft `ScopeOutput` to `_SCOPE_DRAFT_KEY` before `gate_and_revise`, allowing interrupted scope sessions to resume without losing the structured answer.

Proposed

Draft remains non-authoritative

The draft can seed a later approval gate, but task contracts and execution admission should cite the approved scope, not the draft.

Preserved

Draft evidence does not unlock execution

The control plane must not treat draft planning artifacts as proof that implementation can begin.

S3. Gate, Revise, Finalize

Approves the scope and writes durable planning outputs.

Existing

`gate_and_revise` then `_finalize_scoping_outputs`

The scope is reviewed and revised through the hosted gate. On approval, structured scope/project outputs are finalized; fallback behavior writes `scope` and the approved marker.

Proposed

Approved scope becomes contract context

Execution-control modules must treat finalized scope as immutable planning evidence for the feature unless planning is rerun.

Preserved + referenced

Downstream traceability improves

Scope approval remains the same, but implementation contracts can now explicitly point to the scope evidence they enforce.

S4. Structured Recovery Fallback

Recovers model-shaped scope from approved text when needed.

Existing

`_recover_scope_model` fallback

If the gate returns text or only approved scope text is present, the phase attempts to recover a `ScopeOutput`; if recovery fails, it still persists approved text and state scope.

Proposed

Contracts tolerate text-backed scope

Later contract compilation should accept either structured scope or approved text evidence, while marking provenance clearly.

Preserved + explicit

Execution sees provenance, not guesswork

The planning fallback is preserved, but the implementation layer should know whether scope evidence was structured or recovered.

S5. Final Artifact Write

Commits approved scope and project outputs to planning evidence.

Existing

`_finalize_scoping_outputs` or fallback writes

Successful structured approval finalizes scope/project artifacts and approved markers. Fallback approval writes `scope`, `_SCOPE_APPROVED_KEY`, and `state.scope`.

Proposed

Final scope becomes immutable planning source

Future execution state should reference this approved scope through stable evidence links rather than copying it into ad hoc prompts.

Preserved + referenced

Traceability improves without changing scoping

The scoping write remains the same; downstream records get better at pointing back to it.

S6. Broad Planning Handoff

Returns state so broad planning can build shared artifacts.

Existing

Phase returns `BuildState`

`PlanningWorkflow` advances from scoping to broad planning with the approved scope in state and artifacts.

Proposed

Control-plane boundary remains later

No execution-control journal row is created until implementation consumes the approved root DAG.

Preserved

Feature progress stays phase ordered

Scoping remains the first feature boundary and cannot be skipped by the transactional execution architecture.

SF1. Resume And Scheduler Setup

Loads decomposition and planning-control scheduler state.

Control, stage, threads, locks

The phase loads decomposition, derives the effective resume stage, syncs subfeature threads, resets stale background state, and creates locks plus an active task map for subfeature scheduling.

Planning scheduler remains upstream

SF1A. Thread Sync And Stale Background Reset

Normalizes persisted planning threads before new work starts.

Existing

`sync_subfeature_threads` and `_reset_stale_background_state`

The phase syncs thread records against the decomposition, repairs legacy reconciliation status where needed, and clears stale background state before dispatching more planning tasks.

Proposed

Planning-thread repair remains upstream

Execution-control resume repair should not consume these planning thread records as task-attempt or merge-queue state.

Preserved + separated

Resume domains stay distinct

Planning background recovery and implementation crash recovery have different meanings and must remain separate journals.

SF2. Step Dispatch

Runs PM, design, architecture, and test-planning steps.

Existing

`_dispatch_step` chooses per-step runner

Each subfeature step routes to `_run_pm_step`, `_run_design_step`, `_run_architecture_step`, or `_run_test_planning_step`, depending on the current planning-control step.

Proposed

Step outputs remain planning artifacts

PM/design/architecture/test-plan outputs become later contract and verifier context inputs, but this dispatch is not sandboxed product implementation.

Preserved

Planning agents still produce planning outputs

No implementation queue item is created by subfeature planning.

SF2A. Per-Step Locking

Prevents duplicate planning work inside a subfeature step.

Existing

`control_lock`, `step_locks`, `active_tasks`

The subfeature scheduler uses a shared control lock, per-subfeature step locks, and an active task map so concurrent planning steps do not trample persisted planning-control state.

Proposed

Execution leases are separate and stronger

The merge queue and dispatcher should use typed leases/idempotency, not these in-memory planning task maps, when protecting product mutation and checkpoint authority.

Preserved + contrasted

Same concurrency concept, different authority

Planning locks protect artifact authoring; execution leases protect canonical product mutation.

SF3. Interactive Or Background Mode

Supports user-led planning or agent-fill continuation.

Existing

`_ensure_step_mode` and background records

A ready step can run interactively or switch to agent-fill. The phase marks background state as active/running and persists planning control before continuing.

Proposed

Background planning remains restart-aware

The execution-control journal should not replace this planning resume mechanism. It should only consume approved outputs once they are finalized.

Preserved

Mode choice remains planning-local

The control plane must not treat background planning progress as implementation progress.

SF3A. Background Completion Persistence

Records active/running/complete/failed background planning work.

Existing

`set_background_state` and status repair

When background planning finishes or fails, the scheduler clears the active background marker, restores pending status on failure, and persists planning control before continuing.

Proposed

Execution attempts need durable typed rows

Product implementation cannot rely on in-memory task completion or planning-control flags; it needs typed attempts, sandbox manifests, and queue records that survive restarts.

Preserved + informs

The pattern inspires but does not replace the control plane

The execution design borrows restart-awareness, then strengthens it with database-owned idempotency and recovery.

SF4. Dependency-Aware Planning Loop

Launches ready subfeature steps and detects deadlocks.

Existing

`_step_ready`, active tasks, blocked statuses

The phase checks each subfeature's current step, launches ready tasks, marks blocked steps, waits for active tasks, and raises a deadlock if no work can progress.

Proposed

Planning dependencies remain separate

Implementation dependency enforcement later occurs through DAG contracts and regroup overlay validation, not by reusing this planning loop as execution authority.

Preserved + separated

Dependency concepts are reused, not authority

Planning dependency readiness informs intent; execution dependency readiness is proven by typed checkpoint/effective-DAG state.

SF4A. Blocked And Deadlock Handling

Stops planning when no dependency-ready work can progress.

Existing

Blocked statuses and scheduler deadlock error

Steps that are not ready are marked blocked. If no task can launch and no active task can complete, the phase raises a deadlock with the blocked subfeature-step list.

Proposed

Execution deadlocks route through failure taxonomy

Implementation deadlocks should become typed `blocked` or `quiesced` states with route decisions, not generic repair loops or silent phase completion.

Boundary tightened

Blocked means different things by phase

Planning deadlock blocks artifact authoring. Execution blocked state must also preserve sandbox, queue, and checkpoint safety.

Preserved

SF5A. Tail Cascade Targets

Existing

`_cascade_requests` by artifact prefix

Preserved + safer downstream

Preserved + strengthened later

Generates subfeature DAG fragments with failure reporting.

Existing

Round-based parallel `_decompose_workstream`

Workstreams in each execution-order round run in parallel. Failures write `task-planning-blocked`, mirror it, notify the user, and raise before integration review or DAG compilation.

Proposed

Blocked planning remains a planning failure

Control-plane failure routing starts in implementation. Invalid task planning should still stop before execution state exists.

Preserved

Planning blockers do not become product repair

A failed DAG decomposition should not dispatch implementation agents.

TP4A. Blocked Report And Notification

Stops before integration review if DAG fragments are invalid.

Existing

`task-planning-blocked` artifact and `Notify`

Crashed or invalid workstream decompositions produce a blocked report, mirror write, and notification, then raise before DAG integration review or root compilation.

Proposed

Execution remains uninitialized

Because no approved DAG exists, the control plane must not create task attempts, sandboxes, or merge queue rows for this feature.

Preserved

Planning failure stays planning failure

The architecture keeps this fail-closed behavior instead of routing invalid DAG planning into implementation repair.

TP5. DAG Integration Review And Revision

Reviews the integrated DAG before compilation.

Existing

`integration_review` plus optional targeted revision

The ordered decomposition goes through DAG integration review. If revision is needed, targeted revisions patch affected subfeatures before compilation continues.

Proposed

Integration review remains pre-execution

The execution control plane assumes the root DAG already passed this planning integration review before contracts are compiled.

Preserved

Review stays upstream of task contracts

Contracts should be compiled from the reviewed DAG, not from pre-review fragments.

TP5A. Revision-Plan Hooks

Allows DAG gate findings to patch affected subfeatures.

Existing

`_dag_gate_review_hooks` and `targeted_revision`

Integration review and gate review use compiled transforms and revision-plan handlers so DAG findings can be turned into targeted planning revisions before the final root DAG is accepted.

Proposed

Task-definition repair remains pre-execution

Once implementation begins, changing task definitions should require a new approved DAG or explicit overlay, not ad hoc mutation of active execution state.

Boundary tightened

DAG mutation boundary is explicit

The approved DAG can seed execution; unapproved DAG revisions cannot mutate active task contracts.

TP6. DAG Compile, Gate, Persist

Creates the executable DAG consumed by implementation.

Existing

`compile_artifacts`, `interview_gate_review`, `dag` write

The phase compiles the root DAG, gate-reviews it as `ImplementationDAG`, then writes `dag` to Postgres and the artifact mirror via `_persist_approved_root_implementation_dag`.

Proposed

Root DAG becomes typed execution seed

Implementation starts by loading this immutable root DAG, resolving any active typed overlay, compiling contracts, and creating typed execution records.

Boundary tightened

Approved DAG no longer equals runnable agent context by itself

The DAG is still the handoff, but the control plane must derive contracts, workspace snapshots, and queue-safe execution state before agents run.

TP7. Implementation Handoff

Defines the only normal entry point into execution.

Existing

`state.dag` returned to `FullDevelopWorkflow`

After `_persist_approved_root_implementation_dag`, the workflow can advance into `ImplementationPhase`, which loads the root `dag` and starts DAG execution.

Proposed

`ExecutionBootstrap` initializes typed records from the DAG

The new architecture starts by validating the approved DAG, resolving any active overlay, compiling task contracts, and creating typed journal seeds before runtime dispatch is allowed.

Changed

The handoff becomes a typed admission gate

A persisted DAG remains necessary, but no agent runs until the control plane proves workspace, contract, and scheduler readiness.

1. Phase Entry And Quiesce

How a feature enters or resumes implementation.

Existing

Workflow runner -> `ImplementationPhase.execute`

The runner resolves the feature phase. `ImplementationPhase.execute` loads implementation state, handles DAG execution, persists handover, and later runs post-DAG gates.

Current quiesce behavior is preserved: quiesce is a pause boundary, not a phase success.
Proposed

Phase adapter -> `ExecutionControlPlane`

`ImplementationPhase` remains the entrypoint and shim surface, but delegates product-authoritative execution to the control-plane coordinator.

  • State transitions are typed and idempotent.
  • Quiesce propagates without advancing to the next phase.
Changed

Phase behavior stays; authority moves

The phase stops being the owner of every execution decision. It becomes the compatibility adapter around typed services.

2. Scheduling And Effective DAG

Which group/wave should run next.

Existing

`_implement_dag` scans checkpoints

Root `dag`, latest `dag-group:*`, quiesce markers, and active regroup artifacts determine the next group. The root DAG is not overwritten by regroup activation.

Risk: effective-DAG knowledge can be uneven across dispatch, resume, dashboard, and post-test readers.
Proposed

`RegroupOverlayService` + effective-DAG resolver

Typed active overlay state defines effective order for dispatch, resume, checkpoint coverage, rollback-before-first-wave, and post-test readiness. `dag-regroup:*` artifacts mirror it for legacy readers.

Changed

Overlay becomes a typed scheduler input

Scheduler feedback can influence future grouping, but safety constraints still outrank speed: dependency order, write sets, sandbox/merge safety, checkpoint proof, and rollback rules.

3. State And Projections

What is durable authority.

Existing

Postgres artifacts/events are the practical source

`dag`, `dag-task:*`, `dag-verify:*`, `dag-commit-failure:*`, `dag-group:*`, `dag-gate:*`, and events are used for resume, dashboard, supervisor, regroup, and analysis.

Risk: artifact key names can blur attempt evidence, verifier evidence, and checkpoint authority.
Proposed

`ExecutionControlStore` / typed journal

Postgres typed rows are canonical. Legacy artifacts are synchronous compatibility projections with projection links, idempotency keys, hashes, and recovery semantics.

  • No typed success without required projection parity while legacy readers exist.
  • No broad artifact-body scans for default readers.
Changed

Artifacts become views, not the source of every decision

The control plane can reconstruct state from typed rows and explain exactly which row produced each legacy artifact projection.

4. Workspace And Path Authority

Which repo/path is canonical and writable.

Existing

Worktree setup and guards inside implementation

`_ensure_task_worktrees`, worktree registry artifacts, alias pre-dispatch guards, and ACL normalization run inside the DAG loop.

Risk: agent-user writeability, alias drift, and stale path evidence can still surface late in verification or repair.
Proposed

`WorkspaceAuthority` owns this boundary

Canonical repo identity, alias classification, divergent alias fail-closed behavior, ACL/writeability proof, symlink/outside-root checks, dirty generated-output checks, and workspace snapshots are resolved before dispatch.

Metadata-only chmod/chgrp/setgid or identical-content writeability repair may be recorded; product-content mutation waits for merge queue.
Changed

Workspace failures stop before model work

Resolvable alias/ACL classes route deterministically. They do not become broad RCA loops or operator file-copy tasks.

5. Task Contracts And Preflight

What a task is allowed and required to change.

Existing

Task specs plus preflight checks

DAG tasks include file scopes and acceptance context. `_run_dag_group_preflight` catches structural issues around verifier/checkpoint flow.

Risk: no single durable contract owns required, allowed, and forbidden paths across prompts, repair, verify, and commit.
Proposed

`TaskContractCompiler` + deterministic gates

Contracts compile required, allowed, and forbidden paths; acceptance criteria; verification gates; write-set ownership; and merge admission rules.

Changed

Prompts, sandboxes, gates, repairs, and merge share one contract

Task success is no longer inferred only from a model result or latest artifact. Contract evidence becomes a checkpoint prerequisite.

6. Runtime Dispatch

How implementers and repair agents run.

Existing

Agents run against feature repos

`_make_parallel_actor`, runtime policy helpers, and `runner.run(Ask)` dispatch implementers, verifier actors, RCA, repair, and post-DAG gates. Results are persisted as artifacts such as `dag-task:*`.

Risk: successful dispatch can be mistaken for product integration, and agents can mutate canonical repos before merge proof exists.
Proposed

`SandboxRunner` + `RuntimeDispatcher`

Agents run in isolated roots. Runtime dispatch records attempts and runtime failures through a narrow journal facade; it cannot enqueue merge, decide routes, commit, checkpoint, or treat dispatch success as integration success.

Changed

Runtime output becomes patch evidence

`dag-task:*` remains visible, but only as attempt evidence. Canonical repo mutation waits for durable merge queue admission.

7. Verification And Approval

How a candidate becomes mergeable.

Existing

`_verify_and_fix_group` coordinates verifier flow

Raw verifier, expanded lenses, deterministic preflight, repair, and reverify produce `dag-verify:*` artifacts and gate decisions.

Risk: stale context, alias evidence, and product defects can mix inside verifier/RCA loops.
Proposed

`VerificationGraph`, `GateRunner`, `VerificationService`

Candidate manifests, contract ids, workspace snapshots, patch summaries, bounded context packages, raw verifier, lenses, aggregate verdict, `merge_gate`, and `checkpoint_gate` are explicit evidence nodes.

  • Raw `dag-verify:*` is not authoritative alone.
  • Summary-only or stale evidence cannot approve merge.
Changed

Approval becomes graph proof

Merge/checkpoint callers must consume reachable approved graph nodes, not assemble approval from raw artifacts.

8. Failure Routing And Repair

What happens when evidence fails.

Existing

RCA/fix loops live inside implementation flow

`_diagnose_and_fix`, `_single_rca_fix_verify`, direct repair routes, commit-only routes, and artifact repair lanes are coordinated by the implementation phase.

Risk: workflow failures can fall into broad product repair or repeat retry loops.
Proposed

`FailureRouter` decides; bounded executors act

The router classifies and selects the route. `RepairService`, dispatcher, merge queue, workspace authority, and GitService execute only selected bounded actions.

Routes include product repair, contract repair, canonicalization, workspace repair, commit hygiene, verifier/runtime/merge retry, sandbox cleanup, explicit operator-required, or quiesce.
Changed

Failure class controls the next step

Commit hygiene, stale projection, workspace/ACL, runtime-provider, queue recovery, and checkpoint contradiction are workflow classes unless typed evidence proves product defect.

9. Canonical Merge, Commit, Checkpoint

When product files become authoritative.

Existing

Direct git helpers and checkpoint artifacts

`_commit_repos`, `_commit_repos_in_root`, `_commit_group`, commit failure artifacts, verifier approval, and repo cleanliness gate `dag-group:*`.

Risk: commit-only failures happen after verify success and can re-enter repair/reverify paths.
Proposed

`DurableMergeQueue` + `GitService`

Queue lease, claim, apply, rebase/conflict handling, post-apply gates, commit proof, no-dirty snapshot, queue coverage, checkpoint gate, checkpoint body evidence, and idempotent `dag-group:*` projection happen as one recoverable authority path.

Changed

Only the queue mutates product content

Checkpoint projection requires merge proof, commit proof, no-dirty proof, approved gate/checkpoint evidence, checkpoint body evidence, and projection/link success.

10. Post-DAG Business Gates

Feature-level checks after group checkpoints.

Existing

Embedded in `ImplementationPhase.execute`

After all DAG groups checkpoint, the phase runs code review, security, test authoring, QA, integration, final verifier, source-repo push, implementation report, backlog report, and notification.

Proposed

`PostDagGateService`

The same business gates are preserved as feature-level typed gate evidence and `dag-gate:*` compatibility projections. Group checkpoint completion is not final implementation approval.

Preserved + typed

Business logic remains, authority is explicit

These gates cannot be skipped by the new architecture, and failures route through typed verification/router paths before observation can begin.

11. Post-Test Readiness

When observation collection may start.

Existing

Defensive guard checks DAG checkpoints

`PostTestObservationPhase` guards against starting while root DAG checkpoints are incomplete. This was patched after quiesce exposed accidental advancement risk.

Risk: root DAG length alone is not enough when an active effective-DAG overlay exists.
Proposed

`PostTestReadinessGuard`

Observation starts only after effective-DAG completion, post-DAG gate approval, no active queue/repair work, and compatible projections. It never relies on root-DAG length alone.

Changed

Readiness follows effective state

The guard consumes typed completion and effective-DAG resolver evidence, so regroup overlays cannot confuse post-test entry.

12. Observability And Supervisor

How humans and automation inspect the run.

Existing

Dashboard/supervisor read artifacts and events

Bounded summary paths have improved safety, but current visibility still depends heavily on legacy artifact/event interpretation and degraded fallback behavior.

Proposed

Typed snapshots + bounded compatibility fallback

Dashboard, supervisor, and regroup readers use typed snapshots first. Legacy fallback is visibly degraded, bounded, capped, and never hydrates broad artifact bodies.

Supervisor has no executor/control-plane/product mutation authority; bounded audit/dedupe/outbox and explicitly configured guarded bridge actions are the documented exceptions.
Changed

Observability stops being inference-heavy

The operator sees active attempt, failure class, route, queue state, checkpoint proof, and next action from typed evidence rather than heuristic artifact-body scans.

P3. Observation Cycle Resume

Artifact-backed cycle state

Proposed

Cycle state remains, linked to typed readiness

Observation artifacts remain compatibility-visible, but entry and product-changing follow-up work are linked to typed implementation completion and gate evidence.

Restart behavior remains explicit

The observation phase keeps restart-safe cycle artifacts while relying on stronger upstream readiness proof.

P3A. Checkpointed Interview Recovery

Skips re-interviewing after a crash during dispatch.

Existing

`observations-checkpoint:{cycle}`

At the start of each cycle, the phase attempts to parse the structured checkpoint report. If it is valid, it skips collection and resumes dispatch from the recovered observations.

Proposed

Checkpoint recovery remains, dispatch proof strengthens

Observation reports can still be checkpointed, but fixes generated from them should enter the same sandbox/verify/merge proof path as implementation repairs.

Preserved + typed downstream

Observation input recovery is separate from product mutation

Recovering the report does not prove any fix was safely applied.

P4. Observation Interview

Collects user observations, or completes if there is nothing to report.

Existing

`runner.run(Interview(... observation_collector ...))`

The phase asks the user for bugs, missing features, or things that should work differently. If the user says done/nothing to report, the phase exits the cycle loop.

Proposed

User-facing interview stays intact

The collection experience remains the same, but context can point at typed completion/gate evidence instead of inferred artifact state.

Preserved

The user workflow is unchanged

The architecture does not remove the observation interview; it only prevents premature entry.

P4A. Report And Decision Persistence

Writes observations before any fix dispatch occurs.

Existing

`observations:cycle-{n}`, checkpoint, `D-OBS-*`

Collected observations are rendered, serialized, written as the current `observations`, stored in the cycle checkpoint, and any clarification decisions are written as `D-OBS-*` artifacts.

Proposed

Observation input becomes typed repair context

Product-changing observation fixes should record which observation, decision, and upstream completion/gate evidence they are repairing.

Preserved + linked

Input durability remains mandatory

The workflow still persists user-reported issues before attempting fixes; the new architecture links those fixes to mutation proof.

P4B. Observation Context Package

Builds per-cycle context for all observation dispatches.

Existing

`_build_observation_context`

The phase combines the report, cycle number, clarification decisions, prior fix summary, and feature root into context used by each observation fix dispatch.

Proposed

Context includes typed implementation readiness

Observation context should include effective completion, post-DAG gate evidence, workspace snapshot, and current no-dirty proof so fixes cannot operate from stale assumptions.

Changed

Observation fixes inherit the control plane

The dispatch context becomes proof-backed rather than a broad textual summary of prior state.

P5. Observation Fix Dispatch

Handles reported issues through category-specific fix pipelines.

Existing

Priority order, RCA/fix/verify, bounded redispatch

Observations are sorted by priority and dispatched through `_dispatch_observation`. Nonterminal statuses are retried up to `MAX_FIX_ITERATIONS`; `FIXED`, `UNRESOLVED`, `SKIPPED`, `ERROR`, and `BLOCKED` are terminal for that cycle.

Proposed

Fixes respect the same mutation proof rules

Category-specific observation handling remains, but any product mutation must use the control-plane safety boundaries: contracts or bounded repair scope, sandboxed work, verification evidence, merge proof, commit proof, and no-dirty proof.

Changed

Post-test fixes cannot bypass canonical mutation authority

Observation fixes remain user-driven, but product-changing work does not get a side door around sandbox/merge/checkpoint proof.

P5A. Dispatch Failure Classification

Handles prompt-size and runtime exceptions without losing the cycle.

Existing

Prompt-too-long becomes `BLOCKED`; other exceptions become `ERROR`

`_dispatch_observation` exceptions are caught per observation, so one failed fix dispatch records a terminal status instead of destroying all cycle results.

Proposed

Failures route through typed taxonomy

Observation fix failures should use the same router classes as implementation: runtime retry, context-size blocked, product repair, workspace repair, merge conflict, or explicit operator-required.

Changed

Terminal statuses become typed route outcomes

The user-facing statuses remain, but internal recovery becomes more precise and less likely to repeat the wrong fix path.

P5B. Redispatch Budget

Retries only observations that have not reached terminal status.

Existing

`MAX_FIX_ITERATIONS` loop

After each dispatch round, the phase keeps only nonterminal observations and retries them until they finish or the cycle-level iteration budget is exhausted.

Proposed

Retry budget aligns with failure class

The control plane should track retry budgets by failure route so a commit-only, workspace, runtime, or product failure does not consume the wrong retry budget.

Changed

Retries become reason-coded

The bounded loop remains, but route-aware budgets prevent repeated broad repair when deterministic fixes are required.

P6. Cycle Persistence And Completion

Records results and decides whether another observation cycle is needed.

Existing

Artifacts preserve the cycle history

The phase writes `observations`, `observation-cycle-counter`, `observation-decisions`, clears the active checkpoint, and appends `observation-history` before starting another cycle or returning state.

Proposed

Completion remains phase-local, upstream proof remains typed

Observation-cycle artifacts remain readable. Final workflow completion still happens only after the observation loop returns state and shared workflow cleanup runs.

Preserved

History and resume semantics stay visible

The post-test loop remains restart-safe and auditable while upstream implementation readiness becomes more reliable.

P7. Workflow Completion

Returns state only after observation cycles finish.

Existing

No observations means phase complete

When the observation collector returns no observations, or a cycle finishes and no further report is needed, the phase returns `BuildState` to the workflow runner.

Proposed

Completion still follows observation loop

The control plane strengthens readiness and mutation proof, but it does not remove the final user-observation confirmation step.

Preserved

The feature still ends at the same business boundary

The proposed workflow becomes safer and more observable without changing the user-facing definition of a completed feature.
